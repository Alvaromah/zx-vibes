Mejoras identificadas y tarea correspondiente

Este documento traduce el feedback tecnico de varios agentes de codigo a una explicacion clara para decidir que construir, por que importa y como validar cada mejora. La tarea madre es T-20260615-01, Feedback-driven toolkit improvements backlog.

Resumen ejecutivo

Problema central

El toolkit ya permite compilar, ejecutar, inspeccionar y depurar, pero algunos limites aparecen justo donde los agentes necesitan automatizar: estado persistente, snapshots, trazas, graficos, audio y documentacion.

Direccion propuesta

Convertir los flujos repetidos por agentes en comandos estables, JSON parseable, recetas cortas y validaciones que reduzcan scripts temporales.

Forma de ejecucion

No es una unica mega-tarea. Cada mejora debe implementarse como rama o PR pequeño, verificando primero que el feedback sigue siendo cierto en el codigo.

Mapa de tareas

  1. 1. Unificar arranque en run, test y verify.
  2. 2. Hacer seguro el estado y permitir modo solo lectura.
  3. 3. Aclarar frames y mejorar scripting de entrada.
  4. 4. Añadir I/O de snapshots y memoria.
  5. 5. Hacer comandos investigativos stateless y scriptables.
  6. 6. Añadir primitivas de graficos y extraccion de assets.
  7. 7. Mejorar disassembly, xrefs y scans.
  8. 8. Completar docs, assertions y skills.
  9. 9. Mejorar preview, player y ciclo de vida del servidor.
  10. 10. Añadir introspeccion y renderizado de audio.

Trabajo ya resuelto que no debe duplicarse

El feedback mezcla problemas pendientes con contexto historico. Estos puntos deben comprobarse si hay dudas, pero no conviene reimplementarlos sin evidencia de regresion.

Proyectos nuevos ejecutables

El bloqueo inicial de zxs new ya se trato en T-20260614-01: los proyectos generados deben poder ejecutar npm run build y npm test sin buscar bins globales.

Skills locales de agentes

La documentacion de skills de agentes y su copia a proyectos generados ya se completo en T-20260614-02. Las nuevas mejoras deben ampliarla, no rehacerla.

Audio ya enviado

El feedback marca como ya enviados el audio beeper con resampling, los arreglos de teclas pegadas y el sonido de carga de cinta. Solo tocar si el codigo actual contradice ese estado.

Receta 09 beeper

packages/toolkit/recipes/09-beeper-fx/ existe. La mejora real es documentar sonido y referencias, no recrear la receta.

Mejoras detalladas

Cada bloque corresponde a un paquete de trabajo dentro de T-20260615-01. Las areas indican donde probablemente habra que mirar primero, no una lista cerrada de ficheros.

Paquete 1

Unificar el arranque en run, test y verify

Prioridad alta

Por que surge

Un agente observo que el mismo binario podia pasar en run o test, pero fallar en verify. La sospecha es que cada ruta prepara la maquina con un estado inicial diferente.

Problema que resuelve

Evita falsos positivos y falsos negativos en juegos que dependen del arranque ROM, IY, sysvars, interrupciones o RAM inicial.

Solucion propuesta

Auditar run.ts, test-cmd.ts, verify.ts y los helpers de sesion. O bien unificar el boot, o bien mostrar un diagnostico claro cuando una ruta use un modo distinto de forma intencionada.

Aceptacion

Un fixture dependiente de ROM debe comportarse igual en run, test y verify. La ayuda debe explicar cualquier diferencia deliberada.

Areas: toolkit, emulator.

Paquete 2

Estado seguro y modo solo lectura

Prioridad alta

Por que surge

Los comandos persistentes modifican .zxs/state.zxstate. En Windows y entornos gestionados se reportaron fallos EPERM al renombrar el temporal de estado.

Problema que resuelve

Permite que las pruebas de propiedades y los flujos de investigacion no contaminen su estado base ni fallen por operaciones de fichero evitables.

Solucion propuesta

Reintentos o estrategia de reemplazo mas robusta en session.ts, modo global o por comando para no guardar estado, soporte de baseline state save/load y posible ZXS_STATE_DIR.

Aceptacion

Un comando read-only no escribe .zxs/state.zxstate. La documentacion muestra el patron "inyectar estado, ejecutar, comprobar, restaurar baseline".

Areas: toolkit, emulator, reference-docs.

Paquete 3

Semantica de frames y scripting de entrada

Prioridad media-alta

Por que surge

--frames N para en frontera de frame y --keys usa un contador global. Si el juego cambia estado justo despues de HALT, una comprobacion corta puede parecer fallar.

Problema que resuelve

Reduce pruebas fragiles y evita que el agente tenga que contar a mano frames de boot, menu y gameplay para pulsar una tecla en el momento exacto.

Solucion propuesta

Documentar que incluye un frame, promover watchpoints para comprobar escrituras y añadir entrada relativa al inicio del run. Opcionalmente, --until-write, --until-change y estado de teclas por frame en JSON.

Aceptacion

La ayuda y docs incluyen un ejemplo de bucle con HALT. Un agente puede programar entrada relativa sin contar frames historicos.

Areas: toolkit, reference-docs.

Paquete 4

Snapshots y volcado de memoria como ciudadanos de primera

Prioridad alta

Por que surge

Para extraer graficos de un juego existente, los agentes tuvieron que parsear snapshots y convertir salidas hex a binario. Es trabajo generico que se repite en reverse engineering.

Problema que resuelve

Permite inspeccionar registros, RAM, compresion, hardware y layout de paginas sin escribir scripts temporales.

Solucion propuesta

Añadir zxs snapshot info, snapshot ram, snapshot mem, mem dump --out y mem load --bin, empezando por formatos ya soportados y dejando camino a .z80 v2/v3 y 128K.

Aceptacion

Desde un snapshot se puede exportar RAM de pantalla y rangos arbitrarios como binarios. El JSON es estable y parseable.

Areas: toolkit, emulator.

Paquete 5

Comandos investigativos stateless y scriptables

Prioridad media-alta

Por que surge

El flujo actual favorece sesiones persistentes: cargar, poner watchpoint, ejecutar, leer registros, desensamblar. Para agentes eso implica muchos procesos y mutaciones de estado.

Problema que resuelve

Hace viable una investigacion read-only desde un snapshot o estado explicito, con menos latencia y menos riesgo de ensuciar la sesion.

Solucion propuesta

Permitir --z80, --sna, --bin --org o --state en regs, mem, disasm, trace, watch y screen. Añadir watchpoints acoplados a run.

Aceptacion

Un comando puede cargar un snapshot, ejecutar hasta watchpoint y devolver PC, instruccion, direccion tocada, valor, registros y ventana de disasm en JSON.

Areas: toolkit, emulator, assembler.

Paquete 6

Primitivas de graficos y extraccion de assets

Prioridad alta

Por que surge

Los agentes reimplementaron decodificacion de pantalla Spectrum, renderizado 1bpp, atributos, mapas de entropia y deteccion de stride. Todo eso es reutilizable.

Problema que resuelve

Evita scripts desechables para ver sprites, fuentes, SCREEN$, UDGs o bloques graficos. Mejora desarrollo nuevo y extraccion de juegos existentes.

Solucion propuesta

Crear familia zxs gfx: screen, linear, sheet, attrs, font, find y blit-linear.

Aceptacion

Un snapshot o sesion genera PNGs de pantalla, sprites y atributos. gfx find --json devuelve candidatos deterministas para agentes.

Areas: toolkit, emulator, reference-docs.

Paquete 7

Disassembly, referencias cruzadas y busquedas estaticas

Prioridad media

Por que surge

El desensamblador ayuda, pero el agente necesita seguir llamadas, saltos, inmediatos y rutinas que tocan memoria de pantalla sin leer texto manualmente.

Problema que resuelve

Reduce el tiempo de localizar blitters, tablas, llamadas ROM y sitios LDIR/LDDR. El output estructurado permite decidir el siguiente comando automaticamente.

Solucion propuesta

Añadir rangos a disasm, JSON con operandos y targets, simbolos ROM/SLD, zxs scan --opcode, --imm-range y zxs xref --addr.

Aceptacion

Las salidas no requieren scraping. Entradas ROM como RST 0x10, CLS y BEEPER aparecen anotadas.

Areas: toolkit, assembler, reference-docs.

Paquete 8

Docs, assertions, clobbers y skill de reverse engineering

Prioridad media

Por que surge

La informacion existe parcialmente, pero esta dispersa: sonido, vocabulario de assertions, clobbers de ROM, hazards de IX/IY y flujos de asset ripping.

Problema que resuelve

Evita que los agentes aprendan por grep o por fallos en ejecucion. Tambien reduce bugs reales por registros clobbered o strings ROM terminadas en cero.

Solucion propuesta

Crear docs/reference/sound.md, referencia de assertions o test --list-assertions, tabla de clobbers ROM, aviso sobre helpers de impresion y skill/receta de reverse engineering.

Aceptacion

Docs root, toolkit docs y create-package docs quedan sincronizados. Los playbooks generados enrutan a las nuevas referencias.

Areas: reference-docs, toolkit, scaffolding.

Paquete 9

Preview, player y ciclo de vida del servidor

Prioridad media

Por que surge

preview --watch bloquea, puede competir con builds, caer a otro puerto o dejar servidores huerfanos. Tambien falta abrir directamente snapshots o cintas arbitrarias en un player.

Problema que resuelve

Da a los agentes una forma fiable de lanzar, listar, parar y entregar al usuario una URL correcta, sin limpiar puertos con comandos del sistema.

Solucion propuesta

Añadir aviso visible de port fallback, preview --list, --stop, --detach, lock de build/watch y zxs play <file> o preview --load.

Aceptacion

El agente puede gestionar previews sin herramientas externas. Un build obsoleto se avisa. Un snapshot o TAP se abre en navegador con sonido.

Areas: toolkit, emulator, gallery.

Paquete 10

Introspeccion y renderizado de audio

Prioridad media

Por que surge

El JSON actual puede demostrar que hubo actividad beeper, pero no permite escuchar el resultado ni comprobar tono, melodia o segmentos de frecuencia.

Problema que resuelve

Hace testeable el sonido de forma objetiva: no solo "ocurrio audio", sino "la nota aproximada y su duracion son correctas".

Solucion propuesta

Capturar edges con T-states o ciclos, añadir zxs run --wav, exponer edgeTimeline, toneSegments o dominantHz, y revisar fallback de audio.

Aceptacion

Un run headless genera WAV reproducible y metadatos suficientes para afirmar pitch o secuencia de notas en tests.

Areas: toolkit, emulator, reference-docs.

Ajustes menores agrupables

Estos puntos pueden entrar como mejoras pequeñas dentro de los paquetes anteriores o como PRs de baja friccion.

Relajar smoke tests generados para que un segundo sprite no rompa cellsNonBlank max 1.
Añadir nudges de zxs doctor, preview --json y entrega de URL al playbook.
Crear alias watch clear para borrar todos los watchpoints.
Documentar que --bin arranca fresco y que para reanudar tras pokes no debe usarse.
Añadir caveat de screen --text para juegos bitmap o fuentes personalizadas.
Añadir attrNonBlank, coloredCells o documentar alternativas para pantallas solo de atributos.

Validacion esperada

Tipo de cambio Validacion recomendada
Comandos toolkit pnpm --filter @zx-vibes/toolkit typecheck, lint y tests Vitest enfocados.
Snapshots, CPU, memoria o audio Tests relevantes de emulator mas integracion de toolkit.
Docs, skills o assets copiados pnpm --filter @zx-vibes/toolkit run check:docs y pnpm --filter create-zx-vibes run check:assets.
Contratos entre paquetes pnpm run verify desde la raiz.

Criterio de decision

Si una mejora toca comportamiento observable del CLI, JSON publico, snapshots, docs generadas o plantillas, debe validarse tambien desde el punto de vista de un agente: salida estructurada, errores accionables, rutas reproducibles y pocos pasos manuales.
