SPECTRAL_

AI agents for the ZX Spectrum

Un toolchain que permite a agentes de IA (Claude Code, Codex…) escribir juegos de ZX Spectrum 48K de forma autónoma, con un loop de feedback real. Construido sobre zx-generation, un emulador cycle-accurate en JavaScript puro — él mismo 100% generado por un LLM. Un emulador escrito por IA ejecutando juegos escritos por IA.

el loop: ensamblar → ejecutar headless → observar → depurar → iterar

① Qué tenemos (hoy, funcionando y testeado)

Machine headless

Un Spectrum 48K completo corriendo dentro de Node: sin DOM, sin mocks, sin emuladores externos ni sockets. Bota la ROM real hasta © 1982 Sinclair Research Ltd y es determinista (mismo input → RAM bit-idéntica, verificado por test).

zxs CLI

build · run · doctor · bench, todos con --json y exit codes ramificables (0 ok · 1 error · 2 cuelgue · 3 entorno). Inyección de binarios en RAM, presupuestos de frames, --until-pc, carga de TAP.

@zx-vibes/asm backend

Ensamblador embebido por defecto, errores JSON con línea de fuente inlined y símbolos SLD (debug a nivel de fuente). sjasmplus queda como backend opcional.

Screenshots PNG

Framebuffer 352×296 → PNG a 2× (704×592), legible por modelos de visión. Golden test del arranque: la pantalla del boot es un test de regresión.

Rendimiento

6.625 fps — 132× el hardware real, ~463 MHz equivalentes. Cinco segundos emulados cuestan ~40 ms: el loop one-shot del agente es esencialmente gratis.

Calidad

Tests de determinismo, golden screenshot y ensamblar+ejecutar e2e, TypeScript estricto, CI en ubuntu+macos+windows y backend embebido por defecto.

② Qué podemos hacer (el loop del agente, ya operativo)

Un agente de código puede, hoy mismo, sin nada más:

  • 01 Escribir Z80, ensamblarlo y recibir los errores en JSON con sugerencias accionables.
  • 02 Inyectar el binario en RAM y ejecutarlo N frames o hasta una dirección concreta.
  • 03 Leer registros y memoria tras la ejecución (incl. registros sombra que el emulador olvida).
  • 04 Capturar la pantalla como PNG y verla (Claude Code lee imágenes).
  • 05 Ramificar su estrategia según el exit code, sin parsear prosa.
  • 06 Cargar TAPs reales por el camino "auténtico" de cinta.

La filosofía del producto: texto primero, píxeles después. Cada comando devuelve un único documento JSON con direcciones simbolizadas y un campo next que sugiere el siguiente paso — la herramienta enseña su propio workflow.

// zxs build --json: un error que un LLM arregla a la primera
{
  "ok": false,
  "stage": "build",
  "errorCount": 1,
  "errors": [{
    "file": "bad-label.asm",
    "line": 5,
    "message": "Label not found: draw_sprtie",
    "sourceLine": "    call draw_sprtie",
    "hint": "Did you mean 'draw_sprite'?"
  }],
  "next": [
    "fix bad-label.asm:5, then rerun zxs build"
  ]
}

Fase 0 — Walking skeleton

Emulador headless + ensamblador + CLI + screenshot. El loop existe de extremo a extremo.

4–6 días

Fase 1 — Loop de feedback completo

Sesiones con ficheros .zxstate (el agente itera entre invocaciones) · teclas programadas por frame (--keys "60:O*30") · OCR de pantalla contra la fuente de la ROM (zxs screen --text: "ojos" baratos sin gastar tokens de visión) · watchdog de cuelgues: di-halt, pc-loop, rom-error, stack-drift → exit code 2 con diagnóstico y causa probable.

demo: un DI/HALT deliberado sale con código 2 nombrando el PC culpable

5–8 días

Fase 2 — Debugger y tracer

Símbolos SLD: breakpoints por etiqueta o main.asm:42 · watchpoints de memoria · step / step-over · desensamblador · trace con ring buffer y perfil de PCs calientes. Debugging a nivel de fuente desde una shell.

3–4 días

Fase 3 — Servidor MCP

zxs-mcp: una Machine viva entre tool calls, y zx_screen devuelve el PNG como contenido de imagen — Claude literalmente ve la pantalla del Spectrum.

4–6 días

Fase 4 — Conocimiento y scaffolding

zxs new mi-juego: esqueleto que ya funciona (nave + teclado QAOP + loop HALT-synced) con CLAUDE.md · docs destiladas anti-errores-de-LLM (screen interleave, teclado active-low, IM2, attribute clash…) · 12 recetas .asm testeadas en CI — documentación que no puede pudrirse.

🏁 hito interno: Claude Code construye un Pong sin ayuda partiendo del scaffold

continuo

Fase 5 — Hardening y lanzamiento

PRs upstream a zx-generation (exports map · registros sombra en getState() · guards Node-safe) · caracterización ZEXDOC de la CPU · CI Windows · npm publish @zx-vibes/toolkit v0.1.0.

Total estimado hasta una v0.1 publicable: 4–6 semanas a tiempo parcial — con loop de agente usable desde la primera semana (✓ ya cumplido).

El "money shot"

Time-lapse de 3 minutos: Claude Code construye un Pong desde un prompt, y se autocorrige visiblemente — ve estelas del sprite en el screenshot, dice "el orden de borrado XOR está mal", lo arregla, re-ejecuta. Ese momento de autocorrección es todo el pitch.

Galería jugable en el navegador

Hardware real

Un TAP generado por un agente cargado en un Spectrum de gomas, filmado en CRT. Para esta comunidad, ese es el evento de credibilidad — nada lo sustituye.

🏆

AI Game Jam + Season 01

La gente envía prompts, los agentes construyen, la comunidad vota (filtro anti-slop: pasar zxs test + que un humano lo juegue). Y la serie episódica: un juego por episodio, dificultad creciente, publicando también los fracasos — "donde los agentes aún fallan" gana más confianza que un highlight reel.

Defensa ante el escepticismo retro: transparencia total (prompts, transcripts y conteos de iteración publicados), MIT en todo, nunca competir en compos humanas sin declarar, y presentarlo como una nueva rampa de entrada al Z80 — el agente comete los mismos errores que un principiante de 1983.
